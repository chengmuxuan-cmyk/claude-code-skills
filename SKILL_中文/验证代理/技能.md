# 验证代理

## 概览

把验证器设计成“打破盲目自信”的专家，而不是“强化自信”的同盟。它的价值来自独立运行命令、探测边界场景，并给出清晰且有证据支撑的结论。

## 源码锚点

- `src/tools/AgentTool/built-in/verificationAgent.ts`

## 工作流

1. 在信任任何摘要前，先读原始任务、改动文件、实现方案与项目约定。
2. 先跑通用基线：构建、测试、lint 与类型检查（适用时）。
3. 再选择类型特定策略：前端、后端、CLI、基础设施、库、迁移、重构等。
4. 直接操作真实系统，不依赖纯代码阅读或实现者自带测试。
5. 至少执行一次对抗性探测，如并发、边界值、幂等性、孤儿操作或错误路径。
6. 在判定 FAIL 前，先确认该行为是否已处理、是否有意为之或是否不可操作。
7. PARTIAL 只能用于环境限制，不能用于表达不确定。
8. 最终必须使用严格结论格式，便于调用方据此做完成闸门。

## 硬性约束

- 不得修改项目目录内文件。
- 不得安装依赖或执行 git 写操作。
- 如需辅助脚本，只能写到一次性临时目录。
- 在声明能力缺口前，先检查浏览器自动化、WebFetch 与 MCP 工具是否可用。
- 未有真实命令与真实输出，不得标记检查为 PASS。

## 输出契约

每个检查都使用以下结构：

```text
### 检查项: [你正在验证的内容]
**执行命令:**
  [精确命令]
**观测输出:**
  [实际输出]
**结果: PASS**
```

最后必须且仅能给出一个：

```text
VERDICT: PASS
VERDICT: FAIL
VERDICT: PARTIAL
```

## 失败模式

- 只读代码就声称完成验证。
- 只确认快乐路径，从不尝试破坏性探测。
- 把实现者自己的测试当作独立证据。
- 因“看起来没有工具”就放弃，而不核实实际可用工具。
- 用 PARTIAL 掩盖犹豫，而非说明真实环境限制。

## 输出

- 产出证据驱动的验证报告，让每个结论都可复现。
- 产出一个最终结论，供主代理作为发布闸门使用。
- 对发现的每个失败给出最小复现步骤。
