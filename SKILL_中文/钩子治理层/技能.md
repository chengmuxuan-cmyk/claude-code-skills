# 钩子治理层

## 概览

把钩子设计为治理层，而不是执行层本体。钩子应注入策略、附加上下文、修改 MCP 输出或阻止继续执行，但不能破坏底层工具协议。

## 源码锚点

- `src/services/tools/toolHooks.ts`
- `src/services/tools/toolExecution.ts`

## 工作流

1. 先定义生命周期事件：`PreToolUse`、`PostToolUse` 与 `PostToolUseFailure`。
2. 为每个事件定义允许结果字段，如 `updatedInput`、`permissionBehavior`、`additionalContexts`、`blockingError`、`preventContinuation` 与 `updatedMCPToolOutput`。
3. 将前置钩子权限结果交由基础权限系统解析，避免钩子 allow 绕过规则 deny/ask。
4. 将后置钩子副作用归一到独立通道：消息、阻断、附加上下文、继续控制与 MCP 输出变更。
5. 除非产品策略要求 fail-closed，否则钩子取消与钩子错误应可见但非致命。
6. 记录钩子墙钟耗时，便于定位“系统卡住”的真实原因。

## 设计规则

- 保持强类型钩子契约，不要用自由文本承载控制语义。
- 对阻断输出去重，避免同一策略决策在 UI 重复渲染。
- `updatedMCPToolOutput` 仅允许用于 MCP 工具。
- 允许钩子加上下文，但不允许其静默重定义基础执行协议。
- 用专用信号建模“继续执行控制”，不要复用 deny 语义。
- 让钩子错误可观测，同时避免治理层成为单点故障。

## 失败模式

- 把钩子 allow 当作最终 allow，绕过核心权限规则。
- 同一事件同时输出 `blockingError` 与 `hook_blocking_error`。
- 让钩子异常中断整个工具流水线。
- 允许非 MCP 工具通过 MCP 专属路径改写输出。
- 继续执行行为定义不明确，迫使调用方从文本猜测。

## 输出

- 产出每个生命周期事件的钩子契约矩阵。
- 产出治理合并规则，定义钩子与权限系统如何交互。
- 产出非致命错误策略，明确哪些钩子失败要阻断，哪些只提示。
