# 技能工作流封装

## 概览

把技能视为可控工作流包，而不只是 markdown 文档。它必须可发现、可触发、可预算、必要时可隔离执行，并在转录中有明确表示。

## 源码锚点

- `src/tools/SkillTool/SkillTool.ts`
- `src/tools/SkillTool/prompt.ts`

## 工作流

1. 先定义命令身份：名称、描述、`whenToUse`、别名以及是否允许用户直接调用。
2. 对列表文本设置硬上限，确保技能发现足够便宜，可放进主提示词。
3. 当请求明显匹配技能时，将技能调用设为阻塞性要求。
4. 谨慎合并本地技能与 MCP 技能，并过滤掉并非真正技能的 MCP 普通提示。
5. 根据隔离需求、token 预算与流程复杂度，在内联执行与分叉执行间做选择。
6. 对分叉技能，显式准备上下文、流式上报进度，并把结果回写转录。
7. 跟踪使用情况、发现来源与执行上下文，支持后续排序与推荐优化。
8. 当技能已加载或正在运行时，防止重复调用。

## 设计规则

- 将 `whenToUse` 写成高信号触发语言，而非通用宣传文案。
- 优先短描述且保持匹配质量，避免长描述浪费首轮预算。
- 调用语义要明确，避免模型“提到技能但不调用技能工具”。
- 给分叉技能独立 token 预算与消息收集，避免污染主推理路径。
- 为插件与市场技能保留命名空间，避免冲突。
- 在技能注入时添加转录标记，便于阻止递归重入。

## 失败模式

- `whenToUse` 过于模糊，导致无法稳定触发。
- 技能描述无限膨胀，列表成本超过任务本身。
- 文本中提到了技能，但从未调用 Skill 工具。
- 把原始 MCP 提示当作技能暴露。
- 分叉技能执行无进度和结果回报，让调用方陷入黑盒。

## 输出

- 产出技能命令契约，包含名称、描述、`whenToUse` 与执行模式。
- 产出列表预算策略，说明截断与排序原则。
- 产出分叉判定标准，说明何时内联执行、何时隔离到子代理执行。
