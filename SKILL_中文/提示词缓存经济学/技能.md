# 提示词缓存经济学

## 概览

把提示词缓存当作真实系统预算。任何改变前缀、工具集、模型参数或思考配置的因素都会影响延迟与 token 成本，因此缓存复用必须在架构设计起点就纳入。

## 源码锚点

- `src/constants/prompts.ts`
- `src/services/compact/prompt.ts`
- `src/tools/SkillTool/prompt.ts`

## 工作流

1. 列出所有参与缓存键的因素，如系统提示前缀、工具集、模型与思考配置。
2. 按稳定性给提示内容分层，把最稳定内容尽量前置。
3. 用显式边界切分静态与动态内容，避免运行时信息污染缓存前缀。
4. 设计可缓存复用的 summary/compact/分叉代理提示，在需要时提供纯文本与无工具模式。
5. 为技能列表、工具描述与服务器指令设硬预算，防止发现文本占掉主任务预算。
6. 将 MCP 连接状态或实验增量等晚变化数据放到非缓存尾部或增量附件。
7. 当分叉必须共享缓存时，让工具、模型与思考参数与父请求严格对齐。
8. 总结草稿区保持一次性，仅把最终总结内容持久化回上下文。

## 设计规则

- 偏好稳定静态前缀 + 窄动态尾部。
- 对总结代理，若一次无效轮次就会毁掉缓存收益，则应禁止工具调用。
- 激进裁剪发现文本，仅保留提升匹配质量的信号。
- 只有在缓存收益明显时，才牺牲灵活性去严格保持父请求形态。
- 晚到外部状态做增量更新，而非整段提示重建。
- 记录每次缓存未命中原因，使优化工作可聚焦。

## 失败模式

- 构建“静态前缀”时读取会话类型、语言或 MCP 连接状态。
- 总结提示保留工具调用自由，唯一轮次被拒绝工具调用浪费。
- 技能描述无限增长且无预算上限。
- 计划共享缓存的分叉却改了工具或思考配置。
- 可增量附加的信息却被每轮重算。

## 输出

- 产出缓存面地图，列出所有可能打破复用的变量。
- 产出稳定前缀契约，说明分叉与压缩路径必须匹配的请求字段。
- 产出提示预算策略，覆盖技能列表、MCP 指令与总结提示。
