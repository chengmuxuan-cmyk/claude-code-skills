# 上下文卫生系统

## 概览

上下文卫生不只是“窗口满了再总结”，而是持续清洁系统：为工具输出设预算、裁剪冗余历史、分层压缩、从溢出恢复，并让附件与排队命令保持稳定顺序。

## 源码锚点

- `src/query.ts`
- `src/services/compact/`
- `src/services/contextCollapse/`

## 工作流

1. 每轮先做 memory 与技能发现预取，把等待时间隐藏在模型流式输出后面。
2. 尽早应用工具结果预算，必要时用持久化引用替代超大结果。
3. 以固定顺序运行 snip、microcompact、context collapse 与 autocompact，而不是随意混用。
4. 压缩成功后，从新状态重建任务预算跟踪、轮次跟踪与压缩后消息。
5. 在尝试恢复路径前，不提前抛出 prompt-too-long、媒体溢出或 max-output 错误。
6. 在上游暴露失败前，先使用 context-collapse drains 与反应式压缩作为结构化恢复步骤。
7. 每轮结束时收集工具使用摘要、memory 附件、技能附件与排队命令，并做去重。
8. 若能力发生变化，轮次结束后刷新工具池，保证下一轮看到正确工具面。

## 设计规则

- 优先保留最近且高粒度上下文，而非一次性压成粗糙总摘要。
- 为每层压缩定义单一职责：snip 去冗余，microcompact 做局部缩减，collapse 投影轻量视图，autocompact 做全局总结。
- 将恢复路径建模为状态机，避免中间错误过早泄露。
- 在跨压缩边界时保持任务预算连续记账。
- 对附件与排队命令做快照，避免同轮变更引发竞态。
- 允许总结提示使用一次性分析块，但仅将总结内容回写上下文。

## 失败模式

- 恢复尚未尝试就提前暴露中间错误。
- 压缩后复用陈旧 token 计数，导致下一轮决策错误。
- 未做前置预算，任由大工具结果淹没提示词。
- 压缩层顺序错误，丢失本可由更便宜机制保留的细节。
- 实时读取排队命令而非快照，造成竞态轮次行为。

## 输出

- 产出上下文流水线图，展示采样前所有卫生处理阶段。
- 产出溢出、413 与 max-output 场景的恢复阶梯。
- 产出附件与队列策略，定义哪些进入上下文、哪些延后、哪些去重。
