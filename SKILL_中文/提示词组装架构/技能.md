# 提示词组装架构

## 概览

把提示词构建当作流水线，而不是临时字符串拼接。保持长期稳定静态前缀，把易变分段后置注入，并把缓存行为作为显式设计目标。

## 源码锚点

- `src/constants/prompts.ts`
- `SYSTEM_PROMPT_DYNAMIC_BOUNDARY`
- `getSessionSpecificGuidanceSection()`
- `getSystemPrompt()`

## 工作流

1. 盘点所有提示词组成部分，并按“跨会话稳定”与“按轮波动”分类。
2. 将身份框架、基础行为、安全规则与通用格式放在静态区。
3. 将语言偏好、环境细节、技能发现、MCP 指令、记忆、草稿区与 token 预算放在动态区。
4. 增加显式边界标记，让缓存层、调试工具与 API 层在同一点切分。
5. 通过注册表或解析器管理动态分段，避免分散字符串追加。
6. 任何会改变前缀哈希的运行时特性开关都放在边界后，或延迟解析。
7. 将提示分段以数组返回，最后再拼接，便于测试、过滤与 diff。
8. 为分段分配稳定 ID，便于追踪回归、缓存未命中与分段级变更。

## 设计规则

- 静态前缀尽可能长且尽可能稳定。
- 将身份框架、行为策略与工具指引分离，避免互相覆盖。
- 将运行时可变性视为一级风险源，而非实现细节。
- MCP、技能与实验指令后置注入，避免晚到连接打碎缓存前缀。
- 每条可复用规则只保留一个定义来源，防止分段长期漂移。
- 记录每个动态分段依赖的输入，让缓存未命中可解释。

## 失败模式

- 构建静态前缀时读取运行时状态。
- 先生成一个超大字符串，再尝试事后切分或裁剪。
- 让特性开关在“应当稳定”的前缀里膨胀出大量变体。
- 多个 helper 重复追加语气/格式规则，最终互相冲突。
- MCP 或技能指令在无变化时仍每轮重算。

## 输出

- 产出分段映射表，标明每个块的来源、稳定性与缓存归属。
- 产出边界检查清单，说明哪些条件必须留在动态区。
- 产出最小提示构建器，将配置与环境映射为分段数组和最终提示词。
