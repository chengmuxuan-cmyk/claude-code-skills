# 影响范围权限控制

## 概览

权限系统不只是“是/否”闸门，而是分层判断：谁承担风险、动作影响面有多大、即使在自治模式下哪些护栏也必须保留。

## 源码锚点

- `src/utils/permissions/permissions.ts`
- `src/utils/permissions/permissionSetup.ts`
- `src/components/permissions/`

## 工作流

1. 先收集全部规则来源：用户设置、项目设置、本地设置、会话规则与 CLI 授权。
2. 在工具特定权限检查前，先执行基于规则的 deny 与 ask 判断。
3. 将内容特定 ask 规则与安全检查设为不可绕过，防止快速模式跳过。
4. 再应用模式行为，包括 bypass、accept-edits、plan 与 auto。
5. 进入 auto 前先剥离危险 allow 规则，如 `Bash(*)`、`PowerShell(iex:*)` 或 `Agent(*)`，避免绕过分类器。
6. 对可安全沙箱化命令走沙箱快速路径，其余走分类器或显式用户审批。
7. 跟踪分类器来源、原因、成本与重复拒绝，确保系统可在受控条件下“闭合失败”或回退到提示。
8. 显式建模权限持久化，避免临时会话授权与磁盘长期授权混淆。

## 设计规则

- 先问“谁会被影响”，再问“命令技术上能否执行”。
- 为每个 allow/ask/deny 决策保留来源元数据，保证可解释。
- 将宽泛 shell 与子代理 allow 视为高危，因为它们会禁用后续安全评估。
- 即便自动模式，也要对高影响范围动作保留人工闸门。
- 保持 UI 确认文案与后端权限逻辑一致。
- 退出 auto 时恢复被剥离的危险规则，避免静默丢失用户配置。

## 失败模式

- 把工具级 allow 当作无害，意外开启隐藏 YOLO 模式。
- 让 bypass 跳过内容特定 ask 与安全检查。
- 未先剥离危险 allow 就直接进入 auto。
- 只更新内存权限，导致下一轮行为与用户预期不一致。
- 返回不可解释的拒绝信息，用户与开发者都无法定位原因。

## 输出

- 产出决策阶梯图，按顺序列出规则、工具检查、模式、分类器与提示阶段。
- 产出危险规则目录，说明哪些授权会放大影响范围。
- 产出持久化策略，区分临时授权与长期授权。
