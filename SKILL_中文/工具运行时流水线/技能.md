# 工具运行时流水线

## 概览

将一次工具调用建模为完整运行链：解析工具、校验输入、运行前置钩子、解析权限、执行工具、映射结果、运行后置钩子并处理失败。任何阶段都不应静默扭曲协议。

## 源码锚点

- `src/services/tools/toolExecution.ts`
- `src/services/tools/toolHooks.ts`

## 工作流

1. 解析工具标识，并附加 MCP 服务器元数据、安全日志字段与遥测上下文。
2. 执行前先校验输入，并将 schema 错误映射为稳定且对用户/遥测安全的格式。
3. 运行 `PreToolUse` 钩子，使其可加上下文、改输入、请求 allow/ask/deny 或阻止继续执行。
4. 将钩子输出与基础权限系统合并，但绝不允许钩子绕过规则 deny/ask 决策或安全检查。
5. 仅在权限通过后执行工具，并记录耗时、结果与分类后的错误细节。
6. 将工具输出归一为可消费的转录块，让内置工具与 MCP 工具共用统一退出形态。
7. 运行 `PostToolUse` 与 `PostToolUseFailure` 钩子，保持治理逻辑与工具实现解耦。
8. 将中止、认证失败、超时、输入错误与工具错误分开分类，确保重试策略正确。

## 设计规则

- 权限解析必须是显式阶段，而非工具代码里的副作用。
- 钩子是治理层，不是特权执行层。
- 谨慎克隆或回填输入，确保修正后输入可流转且不破坏转录或缓存键。
- 使用遥测安全错误名，不依赖压缩后构造器名。
- MCP 输出变更只允许通过受控字段（如 `updatedMCPToolOutput`）。
- 当执行被拒绝时返回结构化错误结果，让模型知道“被阻断”，不是“丢失”。

## 失败模式

- 把钩子 allow 当作无条件 allow，绕过设置级 deny 规则。
- 用同一通用错误消息混淆 schema 错误与运行时错误。
- 不同工具输出不兼容结果结构，导致下游无法归一。
- 跳过失败钩子，恰在关键失败时失去治理层。
- 报告 MCP 认证失败却不更新连接状态，造成重复坏调用。

## 输出

- 产出阶段图，覆盖校验、前置钩子、权限、执行、后置钩子与失败处理。
- 产出错误分类体系，包含用户消息、遥测标签与各类重试策略。
- 产出合并规则，明确哪些钩子决策可影响权限，哪些不可以。
